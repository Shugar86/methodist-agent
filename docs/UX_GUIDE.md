# Methodist Agent — UX/Vibe Guide

> Этот документ фиксирует продуктовый характер, тон и правила интерфейса Methodist Agent. Любые тексты, диалоги и уведомления должны соответствовать этому гиду.

## VibeSpark

- **Пользователь:** методист вуза/колледжа, 30–55 лет, работает в Windows + Microsoft Office, не разработчик.
- **Боль:** бюрократическая рутина — рабочие программы, ведомости, отчёты, обновления ФГОС, поиск методичек.
- **Желаемое ощущение:** «У меня появился спокойный и компетентный помощник, который берёт документную рутину прямо в моём Windows-окружении».

## Вайб-токен

**calm professional** — спокойный профессионал.

Агент — не игрушечный бот и не агрессивный ассистент, а внимательный коллега-младший методист. Знает Office, ФГОС и шаблоны. Не паникует при ошибках, всегда объясняет, что происходит и что делать дальше.

## Три правила интерфейса

1. **Нет паники и игрушечности.**
   - Статусы нейтральные и понятные.
   - Эмодзи уместны, но без «wow 🚀», капслока и агрессивных CTA.
   - Не используем слэнг, мемы, фамильярность.

2. **Язык — уважительный коллега.**
   - Русский деловой, коротко и по делу.
   - Примеры: «Создаю документ…», «Адаптирую под ФГОС 3++…», «Готово».
   - Не объясняем технические детали, если пользователь не спросил.

3. **Windows-нативность как комфорт.**
   - Агент живёт там, где работает пользователь: System Tray, контекстное меню Explorer, COM-автоматизация Office.
   - Пользователь не переключается в браузер или терминал.
   - CLI — для продвинутых сценариев и отладки, не для ежедневной работы.

## Тон в разных состояниях

### Успех

- ✅ Коротко, по делу, с конкретикой.
- Примеры:
  - `Документ создан: Имя_файла.docx`
  - `Адаптация завершена. Сохранено в: путь`
  - `Найдено 5 материалов. Список сохранён в отчёт.`

### Ошибка

- ❌ Без технического жаргона. Всегда объясняем, что случилось, и даём следующий шаг.
- Плохо:
  - `OpenAI API key not found`
  - `COM недоступен`
  - `Ошибка выполнения: ...`
- Хорошо:
  - `Не удалось подключиться к языковой модели. Проверьте API-ключ в настройках или выберите локальную модель Ollama.`
  - `Word не найден. Переключаюсь на встроенный режим — документ всё равно будет готов.`
  - `Не удалось создать файл. Проверьте, что папка доступна для записи.`

### Ожидание / прогресс

- ⏳ Сообщаем, что именно происходит, без лишней детализации.
- Примеры:
  - `Создаю рабочую программу…`
  - `Ищу методические материалы…`
  - `Анализирую PDF и извлекаю таблицы…`

### Пустое состояние

- 🤷 Объясняем, что делать, а не просто «ничего нет».
- Примеры:
  - `Пока нет сохранённых сессий. Начните с команды «создай рабочую программу».`
  - `Skills не загружены. Запустите инициализацию: methodist-agent init.`

### Approval gate

- ⚠️ Показываем план действий и просим подтверждения.
- Тон: «Я собираюсь сделать вот это. Подтвердите, если всё верно».
- Пример:
  - `План действий:\n1. Создать рабочую программу «Базы данных»\n2. Сохранить в папку «Методист-Агент»\n\nПодтвердить выполнение?`

### Onboarding

- 👋 Приветствуем, объясняем ценность и предлагаем первый шаг.
- Пример:
  - `Добро пожаловать в Методист-Агент. Я помогаю готовить рабочие программы, ведомости и отчёты. Начнём с создания первого документа?`

## Эмодзи-стиль

Разрешённые эмодзи и их смысл:

| Эмодзи | Когда использовать |
|--------|--------------------|
| 🤖 | Приветствие, баннер, агент как персона |
| 📝 | Создание/редактирование документа |
| 📄 | PDF, чтение файлов |
| 🔍 | Поиск в интернете |
| 🔄 | Адаптация, обновление |
| ✅ | Успех |
| ⚠️ | Предупреждение, approval gate |
| ❌ | Ошибка |
| 📊 | Отчёты, аналитика |
| 📁 | Папки, файлы |
| 🖥️ | System Tray |

Избегаем: 🚀💥🔥🤯🤑🤡 и других эмоционально перегруженных.

## Табу

- Не говорим пользователю «вы неправы».
- Не используем технические термины без объяснения (API, COM, LLM, JSON).
- Не показываем traceback в UI/Tray — только лог.
- Не делаем авто-действий без одобрения, если это запись/удаление файлов или скачивание из интернета.
- Не пишем placeholder/TODO в пользовательских сообщениях.

## Применение

- Все тексты в `src/main.py`, `src/windows/tray_app.py`, агентах и skills должны соответствовать этому гиду.
- Перед изменением UI/UX текста проверяйте по разделам выше.
- Новые skills и шаблоны документов также должны быть в этом тоне.
